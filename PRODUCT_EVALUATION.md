# RouteMind 智行策 产品评估

RouteMind 智行策把自然语言目标、POI 质量、口碑热度、营业状态、移动成本和用户场景结合起来，生成可执行路线或候选推荐。当前系统重点验证“本地生活目标到路线决策”的完整链路。

## 产品价值

传统 POI 搜索通常只回答“附近有什么”。RouteMind 进一步回答：

- 用户是想找一家店，还是想安排一条路线。
- 哪些地点适合当前时间、当前位置和服务范围。
- 多个地点之间是否能顺路到达。
- 为什么这些地点值得推荐。
- 目标不清楚时应该追问什么。

## 能力完整性

| 能力 | 当前状态 |
|------|------|
| 自然语言理解 | 支持餐饮、购物、咖啡、酒吧夜宵、商务短程、半日/一日路线等常见表达 |
| 推荐排序 | 综合质量、热度、品牌、实体可信度、营业状态、距离和路线可行性 |
| 路线规划 | 输出时间轴、移动距离、移动时间和地图折线 |
| 组合意图覆盖 | 显式目标优先覆盖，弱顺序可在减少绕行时调整访问顺序 |
| 多轮交互 | 支持会话上下文、偏好和澄清选项 |
| 服务边界 | 明确当前覆盖成都武侯区与锦江区，不跨区域编造推荐 |
| 隐私设计 | 不收集真实身份、消费记录或长期位置轨迹 |

## 后端质量治理

已经补充的关键能力包括：

- 天府广场等地标坐标校正。
- 火锅品牌和正餐实体识别，降低麻辣烫、冒菜、附属服务等误匹配。
- 茶馆、咖啡、公园、商场、超市、酒吧等高频类型的实体纠错。
- 低价值点过滤，包括入口、停车、快递、充电、停业、装修、售楼和普通经营部。
- 显式目标候选保底，避免“超市+小吃”“商场+咖啡”等组合被同大类高分点挤掉。
- 路线可行性约束，检查单段时间、总移动时间、距离和绕行比例。
- 跨出行方式 fallback，在默认步行覆盖不足时尝试更合理的骑行方案。

## 差异化

### 从搜索到决策

RouteMind 不是只列出附近 POI，而是把目标转成可执行方案。例如“太古里附近逛街喝咖啡”会被理解为商场/购物点与咖啡/饮品点的组合路线，而不是只返回咖啡店列表。

### 从评分到综合质量

系统不只看单一评分，还结合口碑热度、品牌识别、商圈密度、营业时间、移动成本和实体特征，降低误分类和低价值候选的影响。

### 从单轮到连续对话

同一会话下，用户可以继续说“那附近换成茶馆”“不要太远”“还有别的吗”。系统会沿用上下文，减少重复输入。

## 当前边界

- 当前在线体验覆盖成都武侯区与锦江区。
- 在线真实评论、实时排队、天气、人流和交通状态尚未接入。
- 当前评分、营业时间和路网来自本地数据资产及适配层；接入线上服务后可以替换为实时数据。
- 组合路线 fallback 在少数远距离购物/超市场景下耗时较高，需要继续引入距离矩阵缓存和异步预计算。

## 扩展方向

- 接入更多城市数据分片。
- 接入线上 POI、评分、评论热度、营业时间和地图路径服务。
- 增强实时排队、天气、人流和交通状态。
- 引入更完整的距离矩阵缓存和路线搜索剪枝。
- 增强用户授权下的偏好管理和反馈学习。

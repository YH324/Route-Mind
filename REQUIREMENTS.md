# 需求文档

> **定位说明**：本文档描述的是**预研阶段**的功能需求，架构设计已预留全国推广的扩展点。当前仅覆盖成都武侯+锦江两区（47,045 POI），作为算法验证和架构预研的测试场。

## 1. 项目概述

智能路线规划系统，基于用户自然语言目标（如"成都半日游，想逛公园和吃好吃的"），自动从海量POI中筛选、排序、编排生成可执行的带时间轴的路线方案。

### 1.1 当前阶段（成都两区预研）
- 自然语言意图解析（时间预算/出行方式/偏好类型）
- 多维度POI评分与排序（Ground Truth质量/类型匹配/距离/多样性）
- 多方案差异化推荐（紧凑高效/休闲慢游/美食探店）
- 路网级实际路径规划（模拟Dijkstra替代直线距离）
- 营业时间可行性校验

### 1.2 未来阶段（全国推广）
- 城市分片按需加载（不预加载全国全量）
- 真实路网API（高德/百度路径规划）
- 真实POI数据API + 真实UGC评论
- 分布式KNN缓存（Redis按城市隔离）
- 高并发微服务架构（K8s + 城市分片Pod）

## 2. 用户场景

### 2.1 游客模式
**画像**：外地来蓉旅游，时间有限，想体验城市精华。
**典型目标**：
- "成都一日游，想看熊猫和吃火锅"
- "带爸妈逛成都，不要太累"
- "成都三天两夜，想打卡网红景点"

**行为特征**：
- 偏好景点、公园、特色小吃、文化体验类POI
- 时间集中在白天（09:00-18:00），晚上可安排夜市/夜景
- 对价格敏感度中等，愿意为体验付费
- 移动半径较大（3-5km），接受跨区移动
- 希望类型丰富（景点+餐饮+休闲），拍照打卡点多

**路线编排逻辑**：
```
类型优先级：景点(1.5) > 公园(1.4) > 火锅/烧烤(1.3) > 小吃(1.1) > 茶馆(1.0)
停留时间：景点50min | 公园40min | 火锅50min | 小吃20min
半径：5000m
购物类上限：≤1个（仅限特产/文创）
时间约束：优先安排白天营业的景点，晚餐安排火锅
```

### 2.2 出差模式
**画像**：商务人士，时间碎片化，围绕酒店或会议地点活动。
**典型目标**：
- "住在春熙路，晚上想快速吃点好的"
- "中午1小时，公司附近有什么推荐"
- "客户晚宴，需要体面一点的中餐厅"

**行为特征**：
- 偏好商务中餐、精品简餐、酒店周边便利设施
- 时间碎片化（午休1-2h / 晚上19:00-22:00）
- 预算中高档（公司报销标准）
- 移动半径极小（1km以内），拒绝长距离移动
- 对营业时间敏感（必须适配会议间隙）
- 不需要景点/游乐园等耗时项目

**路线编排逻辑**：
```
类型优先级：中餐(1.2) > 茶馆(1.1) > 外国菜(1.0) > 按摩SPA(0.9) > 商场(0.7)
停留时间：中餐35min | 简餐20min | 茶馆30min
半径：1000m（硬约束，超距直接过滤）
单段移动上限：15分钟（硬约束）
购物类上限：0个（出差不需要购物）
景点/公园上限：0个
时间约束：严格校验到达时间是否在营业时段内
```

### 2.3 居民模式
**画像**：本地居民，日常休闲或周末放松，熟悉城市。
**典型目标**：
- "周末带孩子在附近逛逛"
- "晚上想找个安静的地方喝茶"
- "家附近有什么新开的火锅店"

**行为特征**：
- 偏好日常餐饮、超市、健身、茶馆/咖啡馆、社区周边
- 周末可全天（10:00-21:00），工作日晚上（18:00-22:00）
- 价格敏感，追求性价比
- 移动半径中等（2-3km），围绕居住区
- 喜欢发现新店（近期开业/高评分但冷门）
- 对排队敏感（高峰期避开）

**路线编排逻辑**：
```
类型优先级：火锅(1.3) > 烧烤(1.2) > 茶馆(1.1) > 公园(1.0) > 健身(0.9) > 超市(0.7)
停留时间：火锅50min | 茶馆35min | 公园40min | 健身60min
半径：2500m
购物类上限：≤2个（日常采购）
新店发现加成：开业<6个月的POI额外+0.3分
排队规避：高峰时段避开排队>4的POI
```

## 3. 功能需求

### 3.1 输入层
- [x] 自然语言目标解析（时间/方式/偏好）
- [x] 用户模式选择（游客/出差/居民），默认游客
- [x] 中心坐标+半径参数
- [ ] **城市定位**（全国推广时从query或GPS解析城市）

### 3.2 数据层（预研→全国）
- [x] POI本体数据加载（当前：本地JSON → 未来：高德/百度API）
- [x] Ground Truth评分生成（当前：规则生成 → 未来：平台真实评分）
- [x] UGC评论生成（当前：规则生成7.75M条 → 未来：美团/点评API）
- [x] 路网图构建与Dijkstra路径计算（当前：模拟路网 → 未来：高德路径API）
- [x] 营业时间数据加载与校验（当前：规则生成 → 未来：平台真实数据）
- [ ] **城市数据分片按需加载**（全国推广必需）
- [ ] **KNN缓存分布式存储**（Redis按城市隔离）

### 3.3 规划层
- [x] 候选POI筛选（半径+可达性）
- [x] 多维度评分（GT质量/类型匹配/多样性/距离）
- [x] 贪心路线构建（带时间预算约束）
- [x] 营业时间可行性过滤
- [x] 购物类比例硬约束
- [x] KNN懒加载缓存图（请求内复用，可配置落盘）
- [x] 用户模式差异化权重（游客/出差/居民）
- [x] 单段移动时间硬约束（出差模式≤15min）
- [x] **会话内上下文记忆**（继承上轮中心点/提到类型/推荐POI，支持清除会话）
- [x] **长期轻画像**（`user_id` 记录偏好、避让、饮食限制，写入本地画像文件，`avoid_tags` 硬过滤，并支持 `/api/profile/clear` 删除）
- [x] **多人对话需求聚合**（`DialogueStateTracker` 聚合说话人、地点、食物、活动、预算、顺序；支持 `goal` 多行自动解析）
- [x] **语义需求匹配**（清淡/不辣/约会/亲子/商务/拍照/久坐等需求标签接入排序）
- [ ] **时间感知路由**（早餐/下午茶/夜宵时段偏好）
- [ ] **天气感知推荐**（雨天推荐室内，晴天推荐户外）
- [ ] **实时人流规避**（高峰时段避开排队严重POI）
- [ ] **路线连贯性**（多条路线间空间/时间衔接）

### 3.4 输出层
- [x] 多方案输出（紧凑高效/休闲慢游/美食探店）
- [x] 时间轴格式化（到达/离开/停留/移动）
- [x] Ground Truth验证指标
- [x] 类型分布统计
- [x] 营业时间标注

## 4. 非功能需求

### 4.1 当前预研阶段
- 生成时间：单条路线<5秒（含Dijkstra，热缓存<3秒）
- 内存占用：峰值<1GB（gt_index精简版+空间索引）
- 可扩展性：架构已预留城市分片接口，支持未来接入真实API
- 隐私合规：不使用个人画像/行为数据，全部基于公开POI信息和聚合统计

### 4.2 未来全国推广阶段
| 指标 | 预研阶段 | 全国推广目标 |
|------|---------|------------|
| 响应时间 | <5s（冷）/ <3s（热） | <2s（热门城市缓存饱满后） |
| 并发QPS | 单机 | 1000+（K8s水平扩展） |
| 内存/城市 | ~500MB | ~2-4GB |
| 数据更新 | 一次性生成 | 实时API + 24h本地缓存 |
| 路网精度 | 模拟（绕行系数1.61） | 真实导航级（实时路况） |
| KNN缓存 | 本地JSON（0.1MB） | Redis Cluster（按城市分片） |

# 数据服务扩展指南

RouteMind 的规划引擎与数据来源解耦。POI、评分口碑、营业时间和路径服务都通过统一适配层进入规划流程，因此可以从当前成都数据资产平滑扩展到更多城市和线上数据服务。

## 扩展目标

- 接入线上 POI 搜索和门店详情。
- 接入评分、评价热度和营业时间服务。
- 接入地图路径规划服务。
- 按城市分片加载数据，避免一次性加载全国资产。
- 使用缓存降低接口延迟和配额压力。

## 架构替换策略

```text
Web 工作台
  -> 应用服务层
    -> 规划与排序引擎
      -> 数据适配层
        -> POI 服务 / 评分口碑服务 / 营业时间服务 / 路径服务
```

扩展数据来源时，核心规划逻辑保持不变，只需要让数据适配层返回统一字段。

## POI 字段映射

| RouteMind 字段 | 说明 | 线上服务常见字段 |
|----------------|------|------------------|
| `poi_id` | POI 唯一标识 | 高德 `id`、平台门店 ID |
| `name` | 门店或地点名称 | `name` |
| `type` | POI 类型 | 高德 `type`、平台类目 |
| `typecode` | 类型编码 | 高德 `typecode` |
| `address` | 地址 | `address` |
| `location` | 经纬度 | `location` / `lat` / `lng` |
| `tel` | 联系电话 | `tel` / `phone` |
| `tags` | 标签 | `tag` / 平台标签 |
| `rating` | 评分 | 平台评分字段 |

## 评分与口碑

RouteMind 推荐排序需要以下口碑信号：

- 总体评分。
- 评价样本量或热度。
- 类型相关维度，例如口味、环境、服务、景观、舒适度。
- 适合场景，例如聚餐、约会、亲子、商务、拍照。

线上服务接入后，可以先使用总评分和评价数量，后续再补充多维口碑和评论语义标签。

## 营业时间

营业时间用于判断“预计到达时是否可用”。接入时建议统一成：

```json
{
  "poi_id": "sample-poi",
  "open": "10:30",
  "close": "02:00",
  "overnight": true,
  "rest_day": "无"
}
```

跨天营业、休息日和节假日营业需要单独处理，以保证路线时间轴可执行。

## 路径规划

路径服务需要返回：

- 两点间距离。
- 预计移动时间。
- 可绘制的路线折线。

字段示例：

```json
{
  "distance_m": 540,
  "time_min": 8,
  "polyline": [[30.657, 104.080], [30.658, 104.082]]
}
```

## 城市扩展

扩展更多城市时建议按城市维护独立资产：

- POI 基础信息。
- 评分与口碑索引。
- 营业时间。
- 路网或路径缓存。
- 语义向量。
- 空间索引和类型索引。

热门城市可预热缓存，低频城市可按需加载。

## 缓存建议

| 数据 | 建议缓存周期 | 说明 |
|------|--------------|------|
| POI 搜索结果 | 12-24 小时 | 降低搜索接口压力 |
| POI 详情 | 24 小时 | 名称、地址、类型较稳定 |
| 评分热度 | 1-6 小时 | 热度可能随时段变化 |
| 营业时间 | 24 小时 | 节假日需单独刷新 |
| 路径结果 | 1-24 小时 | 可按起终点和方式缓存 |

## 合规与授权

接入线上数据前需要确认：

- 地图、POI、评分和评论数据的授权范围。
- 第三方服务使用条款。
- 地图 Logo、数据来源和版权标注要求。
- 用户隐私政策和数据保存策略。
- API Key 和访问 Token 的密钥管理方式。

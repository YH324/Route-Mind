# RouteMind 评委阅读指南

## 一句话介绍

RouteMind 是一个面向本地生活出行的智能路线规划产品。用户只需要输入自然语言目标，例如“春熙路附近想吃火锅”“太古里附近逛街喝咖啡”“九眼桥晚上喝酒顺便找点夜宵”，系统会理解意图、筛选真实 POI、给出路线或候选组合，并解释每个推荐点为什么被选中。

当前数据覆盖成都武侯区、锦江区。系统不会伪装成全国服务：对于广州、重庆、青羊区等暂未覆盖区域，会直接提示不在服务范围内。

## 可以现场演示的能力

1. **知名餐饮推荐**

   输入：`春熙路附近想吃火锅`

   预期：优先推荐春熙路/太古里附近更有识别度的完整正餐火锅门店，例如袁老四、海底捞、小龙坎、大龙燚、谭鸭血等，而不是随机小店或误分类 POI。

2. **组合需求规划**

   输入：`太古里附近逛街喝咖啡`

   预期：系统识别为“商场 -> 饮品”的两段需求，而不是只推荐咖啡。示例结果会包含 WIFC 购物中心和咖啡店。

3. **夜间场景**

   输入：`九眼桥附近晚上喝酒，顺便找点夜宵`

   预期：系统识别为“酒吧 -> 小吃”，自动调整到更合理的夜间起始时间，并过滤“健身 club / 营养 club”等名字误伤。

4. **出差效率场景**

   输入：`天府广场附近出差，1小时内找午餐和咖啡`

   预期：如果严格步行和时间约束下无法构成完整路线，系统会给出“候选组合”，包含中餐和咖啡/饮品，而不是硬凑一条低质量路线。

5. **服务范围拦截**

   输入：`广州塔附近吃晚饭`

   预期：返回服务范围外提示，并说明当前仅支持成都武侯区、锦江区。

## 产品设计重点

### 不是简单关键词匹配

系统会先判断用户到底是在找一个点、两三个点的简单组合，还是完整半日/一日路线。

- “想吃火锅”是单点推荐。
- “逛街喝咖啡”是组合路线。
- “成都半日游”才会生成多方案完整路线。

这样做可以避免把一个简单请求过度规划成一串不相关地点。

### 不编造数据

比赛阶段没有真实 API 和真实点评数据，因此我们使用本地 POI、规则生成评分、营业时间和路网数据来模拟生产环境中的数据结构。系统会尽量把数据接口、模型输出和工程边界做成可替换形态。

对于本地语料缺失的地点，例如武侯祠/锦里周边真实景点条目不足，系统不会生成不存在的景点名，而是返回可用的公园/周边候选，并在文档中说明数据限制。

### 推荐结果可解释

每个 POI 都包含 `recommendation_basis`，面向调试和产品解释。推荐依据包括：

- 质量评分
- 类型权重
- 用户偏好命中
- 估计评价热度
- 门店实体可信度
- 品牌识别信号
- 营业状态
- 与起点距离和移动成本

评委不需要阅读代码，也可以从接口响应里看到“为什么推荐它”。

## 技术实现概览

整体系统分为四层：

1. **Web 工作台**

   前端提供自然语言输入、预选场景、用户模式、地图、路线列表、推荐理由和服务范围公告。

2. **应用服务层**

   `app_service.py` 负责参数规范化、服务范围判断、地标中心推断、会话上下文整合和规划结果包装。

3. **规划与排序引擎**

   `route_planner_v3.py` 是核心模型，当前版本为 `route_planner_v3.8` / `feature_ranker_v1.4`。它负责意图解析、候选筛选、实体类型纠偏、评分排序、路线构建和候选组合降级。

4. **本地数据适配层**

   当前读取本地 POI、评分、类型、路网和营业时间数据。接口设计上对齐未来真实 API，后续接入高德/美团/点评数据时不需要重写核心规划逻辑。

## 工程质量

### 服务范围清晰

当前服务区域写入统一配置，响应中会返回 `service_area`。未覆盖城市或区县会被提前拦截，不会默认返回成都推荐。

### 候选治理

系统会过滤或降级这些常见问题：

- 停业/装修/关闭门店
- 入口、门岗、停车场、快递柜、充电设施
- 民宿房源、酒店会展被误判为餐饮
- 茶饮/棋牌被误判为茶馆
- 商业广场/绿地超市被误判为公园
- 健身 club 或营养 club 被误判为酒吧
- 普通购物服务被误判为超市

### 严格组合顺序

对于明确组合需求，系统使用严格类型匹配：

- “酒吧 -> 小吃”不能被替换成“农家乐 -> 火锅”
- “中餐 -> 饮品”不能被替换成“火锅 -> 中餐”
- “商场 -> 饮品”不能退化成只推荐咖啡

### 自动降级

如果路网、营业时间或移动时间约束过严，系统不会凑无关地点，而是返回“候选组合”，让用户继续筛选。

## 仍然诚实保留的限制

- 当前没有真实用户评价、真实热度、真实排队和真实 API。
- 数据只覆盖成都武侯区、锦江区。
- 武侯祠/锦里周边真实景点 POI 条目不足，系统会返回可用候选，不编造景点。
- 生产系统还需要接入真实 POI、真实营业时间、真实地图路由、缓存服务、日志监控和部署网关。

这些限制没有被隐藏，而是通过服务范围、候选组合降级和文档说明暴露出来，方便后续迭代。

# RouteMind 智行策 评审说明

**项目试用**：[http://124.223.28.124/](http://124.223.28.124/)

## 一句话介绍

RouteMind 智行策是一个面向本地生活出行的智能路线规划产品。用户只需要输入自然语言目标，系统会理解意图、筛选 POI、生成路线或候选推荐，并解释每个推荐点为什么适合当前需求。

当前在线体验覆盖成都武侯区与锦江区。对于暂未覆盖的城市或区县，系统会明确提示暂不支持，不会返回缺少数据支持的推荐。

## 推荐体验

1. **知名餐饮推荐**

   输入：`春熙路附近想吃火锅`

   预期：优先推荐春熙路/太古里附近更有识别度的完整正餐火锅门店，而不是随机小店或误分类点位。

2. **组合需求规划**

   输入：`太古里附近逛街喝咖啡`

   预期：识别为“商场 -> 咖啡/饮品”的两段需求，输出带移动距离和时间轴的顺序候选路线。

3. **夜间场景**

   输入：`九眼桥附近晚上喝酒，顺便找点夜宵`

   预期：识别为“酒吧 -> 夜宵”，匹配夜间营业场景，并过滤与酒吧、夜宵不匹配的点位。

4. **出差效率场景**

   输入：`天府广场附近出差，1小时内找午餐和咖啡`

   预期：优先短距离、高效率和营业可行性，返回可执行的商务短程方案。

5. **服务范围提示**

   输入：`广州塔附近吃晚饭`

   预期：提示当前服务范围为成都武侯区与锦江区。

## 产品设计重点

### 不要求固定输入格式

RouteMind 智行策支持自然语言目标，不要求用户使用固定模板。系统会先判断用户是在找一个地点、两三个地点的组合，还是完整路线。

- “想吃火锅”是单点推荐。
- “逛街喝咖啡”是组合路线。
- “半日游”进入完整路线规划。

### 不编造地点

系统只从当前数据资产中选择真实存在的 POI。对于暂未覆盖地区或数据不足区域，会明确说明服务边界，不生成不存在的门店或景点。

### 推荐结果可解释

每个推荐点都包含推荐依据，包括：

- 质量评分。
- 评价热度估计。
- 用户偏好命中。
- 品牌和门店实体信号。
- 营业状态。
- 与起点距离和移动成本。
- 组合路线中的顺序匹配。

前端会把这些信息转成用户可读的推荐理由和精选口碑摘要。

## 技术实现概览

| 层级 | 说明 |
|------|------|
| Web 工作台 | 自然语言输入、动态目标建议、地图、路线列表、推荐理由和服务范围提示 |
| 应用服务层 | 请求规范化、地标中心识别、服务范围判断、会话上下文整合 |
| 规划与排序引擎 | 意图理解、动态时间预算、候选筛选、质量治理、评分排序和路线构建 |
| 数据适配层 | 统一 POI、评分、营业时间和路径接口，便于扩展线上数据服务 |

## 质量保障

### 服务范围清晰

系统在页面入口和接口响应中展示服务范围。未覆盖城市或区县会被提前识别，不会被替换成成都推荐。

### 候选治理

系统会降低或过滤以下常见错配：

- 停业、装修或关闭门店。
- 入口、门岗、停车、快递、充电等设施点。
- 酒店、会展、普通销售点等与用户需求不匹配的 POI。
- 茶饮被误认为茶馆、商业广场被误认为公园、健身 club 被误认为酒吧等类型偏差。

### 组合顺序严格

对于明确组合需求，系统会保留用户说出的类型顺序：

- “酒吧 -> 夜宵”不会变成“农家乐 -> 火锅”。
- “中餐 -> 咖啡”不会变成“火锅 -> 中餐”。
- “商场 -> 饮品”不会退化成只推荐咖啡。

### 多轮与澄清

同一会话中，系统会沿用上次位置和偏好。“那附近换成茶馆”“还有别的吗”可以继续规划；“随便安排一下”会先返回可点击的方向选择。

## 运行方式

```bash
python run_server.py --warmup
```

访问：

```text
http://127.0.0.1:8000/
```

健康检查：

```text
http://127.0.0.1:8000/api/health
http://127.0.0.1:8000/api/ready
```

## 当前服务边界

- 服务范围：成都武侯区、锦江区。
- 数据范围：当前数据资产覆盖上述区域内 POI、营业时间、评分口碑索引、路网和语义向量。
- 地图能力：路线列表和时间轴可独立展示；地图瓦片依赖网络资源。
- 城市扩展：架构已按城市数据分片和统一数据适配接口设计，可接入更多城市数据服务。

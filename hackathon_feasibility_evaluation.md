# 美团 Hackathon 作品可行性评估：智能路线规划系统

## 1. 项目一句话概述

本项目是一个基于自然语言目标的智能 POI 路线规划系统。用户输入类似“春熙路附近，下午四点想吃火锅”“成都半日游，想逛公园和吃好吃的”这样的自然语言需求后，系统会自动解析意图，从本地 POI、评分、评论、营业时间和路网数据中筛选候选点，并输出带时间轴、移动距离、停留时间和多方案风格的可执行路线。

当前项目处于成都武侯区 + 锦江区预研阶段，覆盖约 47,045 个 POI。虽然当前数据主要为本地清洗和模拟生成，但架构上已经预留了全国推广、真实 API 接入、城市分片和分布式缓存等扩展点。

---

## 2. 项目解决的问题

传统 POI 推荐通常只解决“附近有什么”或“评分高的店有哪些”，但真实用户需求更复杂：

- 用户表达是自然语言，而不是结构化筛选条件。
- 用户需要的是一条可执行路线，而不是零散 POI 列表。
- 路线必须考虑时间预算、营业时间、移动距离、出行方式和类型搭配。
- 不同用户场景差异很大：游客想体验城市精华，出差用户重视效率，本地居民更在意日常可达和新鲜感。
- 在美团场景下，推荐结果还应结合评分、评论、排队、人流、营业状态等平台数据。

本项目的核心价值是把“自然语言需求”转化为“可执行的城市生活路线”。

## 3. 核心功能

### 3.1 自然语言意图解析

系统支持从用户输入中解析：

- 时间预算，例如“半日游”“下午四点”“1 小时”
- 出行方式，例如步行、骑行等
- 偏好类型，例如火锅、公园、茶馆、景点、小吃
- 起始地点或中心点，例如春熙路、天府广场
- 推荐意图，例如单点推荐、简单路线、复杂路线

在产品文档中，系统还规划了通过 LLM 判断用户是“只想找一家店”还是“需要完整路线”，避免用户只是想吃火锅时系统强行规划 6 个点。

### 3.2 多方案路线生成

系统可以输出多种路线风格：

- 紧凑高效：在有限时间内尽可能串联更多 POI，移动路径更短。
- 休闲慢游：减少点位数量，增加停留时间。
- 美食探店：优先满足餐饮偏好，适合“想吃火锅/小吃/茶馆”等场景。

每条路线包含：

- POI 顺序
- 到达时间和离开时间
- 停留时长
- 前一站到本站的移动距离和移动时间
- POI 类型和评分
- 总时间、总移动距离和时间利用率

### 3.3 路网级路径规划

项目不是简单用 Haversine 直线距离估算，而是构建了基于 POI 坐标的模拟路网：

- 47,045 个节点
- 291,460 条边
- 使用 Dijkstra 计算最短路径
- 使用 LRU 和 KNN 缓存加速重复查询

当前模拟路网用于预研阶段验证算法。未来可以保留 `network.get_route_between()` 接口，将内部实现替换为高德或百度路径规划 API，上层路线规划逻辑无需重写。

### 3.4 营业时间校验

系统会根据 POI 类型生成或读取营业时间，并在路线构建时过滤不可达或已打烊的 POI。

例如：

- 景点通常安排在白天。
- 火锅可以安排到晚上。
- 酒吧和夜宵更适合夜间。
- 周一闭馆、跨天营业等情况也可通过营业时间规则表达。

这使路线从“看起来合理”进一步接近“真的能执行”。

### 3.5 三种用户模式

项目定义了三种典型用户模式：

| 模式 | 目标用户 | 推荐偏好 | 半径 | 单段移动上限 | 典型场景 |
|---|---|---|---:|---:|---|
| 游客模式 | 外地游客 | 景点、公园、火锅、小吃、茶馆 | 5000m | 30min | 成都半日游、一日游、打卡 |
| 出差模式 | 商务人士 | 中餐、茶馆、简餐、商务餐厅 | 1000m | 15min | 午休吃饭、会议后晚餐 |
| 居民模式 | 本地居民 | 火锅、烧烤、茶馆、公园、健身、超市 | 2500m | 25min | 周末附近逛逛、新店发现 |

这个设计使系统不是一套固定推荐规则，而是能根据用户场景动态调整搜索半径、类型权重、停留时间、购物上限和移动约束。

### 3.6 语义匹配能力

产品文档中规划了基于 GLM embedding-2 的语义检索能力，目标是让系统理解非字面需求：

- “想吃辣的”可匹配火锅、烧烤、川菜。
- “适合约会”可匹配安静、有氛围、景观位、私密性高的餐厅或茶馆。
- “带孩子玩”可匹配亲子餐厅、科普馆、公园、室内乐园。
- “能坐一下午”可匹配茶馆、咖啡馆、有 Wi-Fi 和插座的店。
- “商务宴请”可匹配有包间、服务好、档次较高的中餐厅。

这部分是项目相较普通关键词 POI 检索的重要创新点。

### 3.7 隐私优先的轻个性化

项目明确不依赖手机号、用户 ID、跨会话轨迹、消费记录、社交关系等敏感个人数据，而是使用低风险信号：

- 当前会话内的用户反馈，例如“不要烧烤”“太远了”
- 当前时间，例如早餐、下午茶、夜宵
- 天气信息，例如雨天推荐室内
- POI 匿名聚合统计，例如排队时间、人流热度
- 匿名群体偏好，例如周末晚上某区域火锅热度较高

这种设计更适合企业内部 Hackathon 评审，因为它兼顾了个性化效果和隐私合规边界。

## 4. 模块拆解

### 4.1 前端模块

路径：`web/`

当前产品文档描述的前端由以下文件组成：

- `index.html`
- `styles.css`
- `app.js`

前端主要负责：

- 接收用户自然语言输入
- 选择中心点、半径、城市等参数
- 调用 `/api/plan`
- 展示路线方案、时间轴、POI 信息和路径结果

### 4.2 Web 服务模块

相关文件：

- `web_app.py`
- `app_service.py`

职责：

- 提供 HTTP 服务
- 暴露 `/api/plan` 路线规划接口
- 暴露 `/api/health` 健康检查接口
- 调用核心规划引擎并封装响应

当前服务形态偏轻量，使用 Python 原生 `ThreadingHTTPServer`，适合 Hackathon 快速演示。

### 4.3 路线规划模块

核心文件：`route_planner_v3.py`

职责：

- 解析用户目标
- 筛选候选 POI
- 计算综合评分
- 应用类型多样性约束
- 应用营业时间过滤
- 应用移动时间和总时间预算
- 构建多方案路线
- 输出时间轴结果

核心评分公式可以概括为：

```python
score = base_score + diversity_bonus - distance_penalty
```

其中：

- `base_score` 来自 GT 评分和标签匹配。
- `diversity_bonus` 用于避免路线里全是同一类 POI。
- `distance_penalty` 用于惩罚移动时间过长的候选点。

### 4.4 POI 与类型推断模块

相关文件：

- `poi_loader.py`
- `ugc_type_profiles.py`
- `output/type_index.json`

职责：

- 加载 47,045 个 POI。
- 清洗名称和基础字段。
- 基于名称、tag、typecode 推断 POI 类型。
- 对部分错误类型做启发式修正。

项目中定义了约 36 种 POI 类型，并为不同类型配置不同评分维度。

### 4.5 UGC 与 GT 评分模块

相关文件：

- `ugc_v4_xl_fast.py`
- `ugc_type_profiles.py`
- `ugc_groundtruth_v4_xl.json`
- `output/gt_index.json`

职责：

- 在预研阶段规则生成 775 万条评论。
- 按 POI 类型生成差异化评论和评分。
- 将大体积评论聚合为轻量 GT 评分索引。

当前 `ugc_groundtruth_v4_xl.json` 约 1.97GB，不适合每次规划直接加载，因此项目使用约 2MB 的 `gt_index.json` 作为规划阶段的评分输入。

### 4.6 路网与路径模块

相关文件：

- `road_network.py`
- `generate_road_network.py`
- `poi_knn_graph.py`
- `chengdu_road_network.json`
- `output/poi_knn_cache.json`

职责：

- 从 POI 坐标生成模拟路网。
- 使用 Dijkstra 计算最短路径。
- 使用 KNN 懒加载缓存 POI 间路网距离。
- 为路线规划提供真实感更强的移动距离和移动时间。

该模块是项目从“POI 推荐”升级到“路线规划”的关键。

### 4.7 营业时间模块

相关文件：

- `generate_business_hours.py`
- `poi_business_hours.json`

职责：

- 按类型生成营业时间。
- 支持跨天营业、休息日、峰值时段等字段。
- 在路线规划中判断 POI 到达时是否营业。

### 4.8 LLM 与语义模块

相关文件：

- `llm_clients.py`
- `semantic_search.py`
- `output/poi_embeddings.npy`
- `output/poi_embedding_ids.json`

职责：

- 对接女娲平台或 GLM API。
- 支持 OpenAI / Anthropic 风格协议。
- 进行意图分类。
- 生成或加载 POI 向量。
- 支持语义检索和语义加权。

这部分能增强系统对“清淡”“约会”“商务”“亲子”等隐式需求的理解。

### 4.9 配置与扩展模块

相关文件：

- `config.py`
- `.env`
- `.env.example`

职责：

- 管理 API Key、端口、候选池大小、类型配额、语义增强权重等配置。
- 支持环境变量覆盖。

| 数据 | 规模 | 当前来源 | 未来替换方向 |
|---|---:|---|---|

| 空间索引 | 830 个网格 | 500m 网格 | 按城市重建 |

预研阶段总数据量约 2GB，其中评论原始数据占比最大。实际规划阶段使用轻量索引，避免每次加载大文件。

## 7. 创新性分析

### 7.1 从“店铺推荐”升级为“可执行路线”

普通推荐系统通常只给出 POI 列表，而本项目输出的是带时间轴的路线。它同时考虑：

- 当前时间
- 总时间预算
- POI 停留时间
- 点到点移动时间
- 营业状态
- 类型搭配
- 路线利用率

这使推荐结果更接近真实用户决策。

### 7.2 路网距离替代直线距离

很多原型系统使用经纬度直线距离估算移动成本，但城市路线中直线距离经常失真。本项目用模拟路网 + Dijkstra 计算路径，虽然不是生产级真实道路，但已经验证了“路网级规划”的工程链路。

未来切换到高德路径 API 时，上层规划引擎无需大改。

### 7.3 语义需求匹配

项目不仅识别“火锅”“茶馆”这类显式类型，也规划了对“清淡”“浪漫”“亲子”“商务宴请”“拍照好看”等隐式需求的理解。

这对美团场景很有价值，因为用户真实表达往往是场景型和感受型，而不是标准商户分类。

### 7.4 三种用户模式

游客、出差、居民三种模式体现了同一城市空间中的不同使用逻辑：

- 游客愿意走远一点，追求体验丰富。
- 出差用户时间碎片化，移动距离必须短。
- 居民用户关注日常、性价比、新店和排队情况。

这种模式化设计比单一评分排序更贴近真实业务。

### 7.5 隐私友好的个性化

项目将隐私边界写得比较清楚：不依赖个人身份、消费记录、长期轨迹或跨会话画像，而是使用会话内反馈、时间、天气、匿名人流和聚合偏好。

这使方案在美团这样的真实平台环境中更容易通过合规评审，也降低了 Hackathon 方案落地时的争议。

### 7.6 城市分片和缓存架构

项目没有假设一次性加载全国数据，而是采用城市分片思路：

- 单城市按需加载
- 热门城市预缓存
- KNN 缓存按城市隔离
- 未来可迁移到 Redis Cluster
- 网关按城市路由到不同服务实例

这说明项目不仅关注 demo 效果，也考虑了后续扩展成本。

## 8. 可行性评估

### 8.1 技术可行性：较高

当前系统已经具备完整链路：

- 有 POI 数据。
- 有评分数据。
- 有营业时间数据。
- 有模拟路网。
- 有规划引擎。
- 有 Web API 设计。
- 有前端展示方向。

虽然部分数据是模拟生成，但核心算法链路已经能跑通，适合作为 Hackathon 原型展示。

### 8.2 数据可行性：中等偏高

当前使用本地清洗和模拟数据验证方案，数据覆盖成都两区，足以展示原型能力。

未来接入真实业务数据时：

- POI 本体和营业时间替换成本较低。
- 路径规划 API 替换成本较低。
- 真实评论和 UGC 质量数据替换成本中等，需要平台数据或商务合作。
- 语义标签体系需要数据标注、评论抽取和人工校准。

整体来看，数据链路可行，但生产化效果高度依赖真实美团商户数据、评分数据、排队数据和评论数据。

### 8.3 产品可行性：较高

该项目适合以下美团业务场景：

- 到店餐饮推荐
- 本地生活路线推荐
- 城市周末计划助手
- 酒旅用户目的地玩法推荐
- 商圈内吃喝玩乐组合推荐
- 团购套餐外的周边联动推荐
- “吃完去哪儿”“附近还有什么”这样的连续推荐

它可以作为美团 App 内的智能助手能力，也可以作为商圈页、目的地页、酒店页、火锅/茶馆/景点频道的增强推荐模块。

### 8.4 工程可行性：中等

Hackathon 展示阶段，当前 Python + 本地 JSON + Web 服务足够。

生产化需要补充：

- API 客户端真实接入
- 分布式缓存
- 城市分片加载
- 服务监控和超时降级
- 真实路况和路径 API 配额管理
- 数据更新机制
- 推荐结果可解释性
- A/B 测试平台

所以工程可行，但需要明确从 demo 到生产系统的分阶段路线。

### 8.5 合规可行性：较高

隐私文档明确了数据禁区和可用信息池，整体设计偏保守：

- 不收集手机号、身份、消费记录。
- 不使用跨会话历史轨迹。
- 不建立强个人画像。
- 会话上下文可销毁。
- GPS 位置需要用户主动授权。
- 聚合信号要求匿名且 N≥100。

这对 Hackathon 评审是加分点。

## 9. 当前限制与风险

### 9.1 数据真实性不足

当前 UGC、GT 评分、营业时间和路网都带有模拟成分。它们能证明算法链路，但不能完全证明真实业务效果。

建议在展示时明确口径：当前是预研和架构验证，未来可替换为美团真实数据。

### 9.2 POI 类型质量存在误差

文档中提到 `type_index.json` 存在类型标注错误，例如酒店、小吃、服饰等被误分。虽然已有启发式修正，但语义匹配和路线规划最终仍依赖类型质量。

生产化前需要重新清洗类型体系，并结合商户类目、评论关键词和人工抽检。

### 9.3 LLM 和 embedding 依赖外部服务

意图分类和语义检索依赖 GLM API 或女娲平台。网络异常时需要降级策略：

- 意图分类失败时回退为默认复杂路线。
- 语义检索失败时回退为关键词匹配。
- API 超时时避免阻塞主规划链路。

### 9.4 路径 API 成本与配额

如果未来替换为真实高德路径 API，大规模 POI-POI 路径查询会产生较高调用量。

KNN 缓存、热门区域预计算和城市分片是必要设计，否则成本和延迟会成为瓶颈。

### 9.5 多人对话和长期记忆存在隐私挑战

下一代交互文档中提到长期记忆和多人对话理解，这些能力很有想象力，但也会引入更高合规成本。

如果作为 Hackathon 主线，建议优先展示“会话级记忆”和“语义匹配”，谨慎表达“长期个人画像”。

## 10. Hackathon 展示建议

### 10.1 推荐展示主线

建议将项目包装为：

> 美团本地生活智能路线助手：把用户一句自然语言需求，转成可执行的吃喝玩乐时间轴。

核心演示可以选 3 个场景：

1. 游客：“成都半日游，想逛公园和吃火锅”
2. 出差：“春熙路附近，晚上 1 小时快速吃点好的”
3. 居民：“周末附近逛逛，想找新开的火锅店，不想排队”

通过三组输入展示不同模式下半径、类型、时间和路线结果的变化。

### 10.2 推荐强调的评审亮点

- 不是单点推荐，而是时间轴路线规划。
- 不是直线距离，而是路网级距离和 Dijkstra。
- 不是纯关键词，而是语义需求匹配。
- 不是固定规则，而是游客 / 出差 / 居民多模式。
- 不是侵犯隐私的画像推荐，而是匿名、聚合、会话隔离的轻个性化。
- 不是只能做成都 demo，而是预留了城市分片和真实 API 替换路径。

### 10.3 建议弱化的部分

- 不要过度强调当前 775 万条评论的真实性，因为它是规则生成。
- 不要把长期记忆作为当前已完全落地能力，容易引发隐私追问。
- 不要声称当前模拟路网等价于真实导航，只说它验证了路网规划接口和缓存机制。

## 11. 结论

该项目作为美团 Hackathon 作品具备较强可行性和展示价值。

它的优势在于：选题贴近美团本地生活核心场景，功能链路完整，技术模块清晰，兼顾语义理解、路线规划、营业时间、路网距离和隐私合规。即使当前数据仍处于模拟和预研阶段，项目也已经形成了从 demo 到生产化的合理演进路径。

综合判断：

| 维度 | 评价 |
|---|---|
| Demo 完整度 | 高 |
| 技术创新性 | 较高 |
| 美团业务相关性 | 高 |
| 数据落地难度 | 中等 |
| 工程生产化难度 | 中等 |
| 隐私合规风险 | 较低 |
| Hackathon 推荐度 | 高 |

如果后续继续推进，建议优先完成三件事：

1. 把三种用户模式真正接入 `route_planner_v3.py` 的评分和约束逻辑。
2. 强化语义匹配，让“清淡、约会、亲子、商务”等需求能明显影响推荐结果。
3. 做一个稳定的前端演示页，用三组典型 query 展示路线差异和时间轴效果。

# RouteMind 智行策 产品说明

RouteMind 智行策是一个面向本地生活场景的智能路线规划系统。用户用自然语言描述目标，系统自动识别需求、筛选 POI、生成时间轴路线或候选推荐，并给出清晰的推荐依据。

**项目试用**：[http://124.223.28.124/](http://124.223.28.124/)

## 产品定位

RouteMind 解决的是“如何把一个本地生活目标转成可执行方案”的问题。它不是普通 POI 搜索，也不是固定模板，而是把用户意图、地点质量、口碑热度、营业状态、移动成本和场景偏好组合成可执行路线。

当前在线体验覆盖成都武侯区与锦江区。产品面向多城市设计，后续接入其他城市的 POI、评分口碑、营业时间和路径服务后，可以提供同类体验。

## 典型场景

| 用户表达 | 系统输出 |
|------|------|
| 春熙路附近，下午四点想吃火锅 | 火锅候选列表，优先识别更像正餐火锅、有品牌或商圈信号的门店 |
| 太古里附近逛街喝咖啡 | 商场/购物点与咖啡/饮品点组成的可执行路线 |
| 九眼桥晚上喝酒，顺便找点夜宵 | 自动调整夜间时段，生成酒吧与夜宵组合 |
| 出差 1 小时内找午餐和咖啡 | 更短移动距离、更高效率的商务路线 |
| 随便安排一下 | 返回可点击方向进行澄清，而不是强行生成低确定性路线 |

## 推荐依据

每个推荐点会携带结构化依据和用户可读摘要，包括：

- 质量评分和口碑热度估计。
- 类型命中和语义需求匹配。
- 实体可信度，例如火锅是否为正餐门店、商场是否为商场本体、超市是否不是商超内小店。
- 营业状态和预计到达时段。
- 起点距离、路线移动时间、绕行比例和总移动预算。
- 组合路线中的顺序覆盖、弱顺序重排和可行性说明。

## 模型与规划策略

RouteMind 的后端模型链路分为六层：

| 层级 | 作用 |
|------|------|
| 意图理解 | 区分单点推荐、简单组合路线和完整路线；大模型可用时参与模糊表达理解 |
| 候选召回 | 按服务范围、起点和用户显式目标召回 POI，避免明确目标被候选截断挤掉 |
| 实体治理 | 修正高频类型误分，过滤停业、装修、入口、停车、快递、充电等低价值点位 |
| 排序模型 | 综合评分、热度、品牌、商圈、营业、语义、实体可信度和移动成本 |
| 路线优化 | 用路网或距离图检查移动时间、距离、绕行比例和总移动预算 |
| 路线评审 | 对候选路线做 listwise 比较；外部大模型可参与评审但不能新增或编造地点 |

其中，规则类逻辑只作为安全边界和质量约束使用，例如服务范围、营业可行性、低价值点过滤、路线移动上限。真正的推荐排序由多信号特征和路线可行性共同决定，不使用固定门店清单。

## API 概览

### POST `/api/plan`

```json
{
  "goal": "太古里附近逛街喝咖啡",
  "session_id": "sample-session",
  "user_id": "sample-user",
  "city": "chengdu",
  "center_lat": 30.65335,
  "center_lng": 104.08126,
  "radius": 3000,
  "user_mode": "tourist"
}
```

响应包含：

- `variants`：路线方案或单点候选。
- `route` / `recommendations`：POI、时间、移动距离、移动时间和地图折线。
- `recommendation_basis`：结构化推荐依据。
- `review_summary`：面向用户的精选口碑摘要。
- `interaction`：会话、澄清、多轮上下文信息。
- `service_area`：当前服务范围说明。
- `performance`：加载与规划耗时。

## 服务边界

当前在线体验覆盖成都武侯区与锦江区。暂未覆盖城市或区县会被明确拒绝，系统不会把成都数据替代到其他城市，也不会生成数据中不存在的门店。
